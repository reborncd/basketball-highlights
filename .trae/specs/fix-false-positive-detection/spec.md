# 篮球进球检测优化 - 解决假阳性和重复检测

## Overview
- **Summary**: 优化现有的篮球进球检测算法，解决假阳性（误报）和重复检测问题，提高检测准确率。
- **Purpose**: 当前算法存在明显的假阳性检测（把没有进球的情况误识别为进球）和重复检测（同一个进球被多次识别），需要通过更严格的状态机和轨迹验证来解决这些问题。
- **Target Users**: 篮球视频分析用户

## Goals
- 减少假阳性检测率到5%以下
- 完全消除重复检测
- 保持合理的检测召回率（不明显降低真实进球的检测率）
- 提供更详细的调试日志用于问题分析

## Non-Goals (Out of Scope)
- 不重新设计整个检测架构
- 不引入新的深度学习模型
- 不添加视频编辑功能
- 不修改UI界面

## Background & Context
当前实现了基于griftt/ball-yolo的三阶段区域判定法，但存在以下问题：
1. 假阳性：篮球在篮筐附近晃动或经过其他路径时会被误识别为进球
2. 重复检测：同一个进球在冷却时间内可能被多次触发
3. 缺少轨迹连续性验证：只检查时间窗口，不验证篮球是否真的有从高位到进球区的连续轨迹

## Functional Requirements
- **FR-1**: 实现完整的进球状态机，跟踪整个进球过程
- **FR-2**: 添加篮球轨迹连续性验证
- **FR-3**: 优化冷却时间逻辑，确保同一个进球只检测一次
- **FR-4**: 增加更详细的调试日志，记录检测决策过程
- **FR-5**: 增加进球确认机制，需要满足多个条件才判定为进球

## Non-Functional Requirements
- **NFR-1**: 检测速度不能降低超过10%
- **NFR-2**: 内存使用量保持在合理范围内
- **NFR-3**: 代码可维护性良好，易于理解和调试

## Constraints
- **Technical**: 必须基于现有的PyQt5和OpenCV技术栈
- **Business**: 需要在1-2天内完成改进
- **Dependencies**: 保持与现有项目结构的兼容性

## Assumptions
- 篮球轨迹在真实进球时应该有明显的从高位到进球区的下降过程
- 冷却时间设置为3-5秒应该足够防止重复检测
- 现有的篮筐检测和篮球检测模块基本可靠

## Acceptance Criteria

### AC-1: 实现完整的进球状态机
- **Given**: 篮筐已校准完成
- **When**: 篮球在不同区域之间移动
- **Then**: 系统能够正确跟踪状态转换（等待→高位区→触框→进球区→确认）
- **Verification**: `programmatic`
- **Notes**: 状态机应包含明确的状态转换逻辑

### AC-2: 添加轨迹连续性验证
- **Given**: 篮球进入进球区
- **When**: 检查之前的轨迹
- **Then**: 必须验证篮球确实从高位区或触框区移动到进球区，且轨迹是向下的
- **Verification**: `programmatic`
- **Notes**: 需要检查最近10-20帧的轨迹

### AC-3: 完全消除重复检测
- **Given**: 检测到一个进球
- **When**: 在冷却时间内再次出现类似情况
- **Then**: 系统不会重复触发进球检测
- **Verification**: `programmatic`
- **Notes**: 冷却时间可配置，建议5秒

### AC-4: 增加详细的调试日志
- **Given**: 正在进行检测
- **When**: 做出任何检测决策
- **Then**: 系统会输出详细的调试信息，包括为什么接受或拒绝某次检测
- **Verification**: `human-judgment`
- **Notes**: 日志应该包含时间戳、位置信息、决策理由

### AC-5: 降低假阳性率
- **Given**: 篮球在篮筐附近但没有真正进球
- **When**: 进行检测
- **Then**: 系统不会误判为进球
- **Verification**: `human-judgment`
- **Notes**: 需要通过实际视频测试验证
